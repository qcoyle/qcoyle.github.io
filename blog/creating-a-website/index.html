Building a Website with no Front-end Experience

Have you ever clicked the Chrome “Developer Tools” button and wondered what all that means? Been curious about terms like elements, Javascript, and React? For many years I have. I had taken intro computer science classes in college and have basic familiarity in the Java and Python projects completed in those. But never the front end. It seemed absurd that being on the web is my second-highest source of time use (after sleeping) and yet I knew nothing about it. There are 1.88 billion active websites. Creating a personal website seemed like the perfect way to learn more about web development.

As a bonus, I’d have something tangible to show others. And plus how hard could it be?

As it turns out, there’s a lot out there. The most efficient way to quickly and cheaply get a website up would have been to use web creation tools like Wordpress or Google Sites. I have heard you can get a simple site up in a matter of minutes and create a beautiful personal or company website in under a day. However I wanted to build my own website rather than be tied to the templates and controls of website building tools. The reasons were:

  • To learn more about web development. Gain basic knowledge of CSS, HTML, Javascript, cloud hosting, and other tools I didn’t yet know about
  • Have a unique design layout
  • Have a site with dynamic web pages
  • Incorporate my own projects and data analytics seamlessly into the website
  • Options for custom forms and buttons - collect and manipulate data from users
  • Options for comments and interactive blog posts
  • A wiki to post my own documentation
  • Have a private password-protected portal for personal notes and projects in progress

The above listed requirements ruled out web creation tools. As such the remainder of this post describes my navigation between the swaths of articles, blogs, posts, and documentation that exist for those looking to build a website. I do not go into explaining many of the terms used - hence the aggressive hyperlinking.

When it comes to getting a url and making a website, Codecademy has some great introductory tutorials - Learn to Make a Website and How to Deploy a Website (paid). I completed those rather quickly. However it became apparent that the products of these courses are better-suited for blog sites. In particular, the commonly-used Jekyll site generator (integrated with GitHub pages) was difficult to modify beyond their basic templates. It didn’t provide much more flexibility than the web creation tools. Trying to modify the Jekyll code and create my own themes was a failure. Maybe it would have been easy for a Ruby developer. Nonetheless it was cool to actually have my own url and an accompanying git repo to go along with it. With Jekyll+GitHub pages not going to cut it, the options were either to write everything from scratch (i.e. using HTML and Javascript/Ruby) or to use a Web Framework.

From Scratch or a Web Framework?

This decision took more research than expected. The thought of considering writing a site from scratch seemed absurd at first. Why reinvent the wheel? There are packages out there made and perfected by brilliant front-end developers. Bootstrap is one of them and it seemed to be the way to go for a beginner. This is because it has:

  • A gentle learning curve
  • Good community support
  • Good browser compatibility
  • Good desktop/mobile compatibility automation
  • Extreme popularity - a nice vote of confidence. But also a con

The decision was now whether to build the website from scratch or to use Bootstrap. At this point, I realized that even if I were to use Bootstrap I would need to have some basic knowledge of HTML and CSS. So before making the decision, I took the HTML, CSS I, and CSS II Codecademy courses.

  • Angular JS
  • jQuery

  • React

Indeed, learning how to make dynamic web pages is the primary goal of my building this site. It’s nice to have a foundation to build on while striving to hit that point.

Greatest difficulty:

  • Staying focused. Keeping productive on what’s most important amidst all the tutorials, stackoverflow posts, and features available. This post started as a private brain dump to keep me focused. It has been immensely helpful in doing so

Greatest surprises:

  • I only spent ~10% of project time writing code. Designing, writing requirements, and reading articles took the other 90.
  • HTML and CSS themselves are super logical and I found them easy to get the hang of. It’s the organization that was hard

What I learned:

  • Basics of: HTML, CSS, web page structure, domains, and cloud hosting
  • When reading about or taking courses on new topics, concurrently add projects to your site that help solidify the knowledge from those topics
  • You should design your site for mobile devices first (mobile-first approach). I did not do this, regretfully
  • Deploy often. If you aren’t deploying changes to your site at least every day you work on it, try working on a simpler feature
  • Have organized concept notes that are easily accessible. I found this to accelerate my learning of new concepts
  • HTML/css files take up almost 2x as much space when saved with formatting!
  • If you hyperlink to a folder, the .index html file is what’s read by default
