Git Terminology

Git in a sentence

  • Industry standard version control system for software

3 parts of a git project

  • Working directory: Where you do all the work. Create, edit, delete, and organize

  • Staging area: List changes made to working directory
  • Repository: Git permanently stores changes as different versions

HEAD

  • The commit you are currently on
  • Usually, the most recently-made commit is the HEAD commit

Index

  • Your staging area

Purposes of a staging area

  • Split up one large change into multiple commits after the fact (because you were in the zone)
  • Avoids having to do one huge commit at once
  • Instead, stage specific files selectively while testing that things still work. Then commit them
  • Helps during reviews of a complex commit
  • During code reviews, it helps you check off individual changes
  • You can then concentrate better on files not yet staged
  • Helps with merges
  • During a merge, changes that merge cleanly are updated in staging area and working directory. Changes that caused a conflict (and only those changes) will show up when you do a diff. You can then focus on these issues
  • Helps you do small changes on the fly
  • Not the best practice, but if someone needs you to do a quick change that’s only a line or two (and completely separate from the work you’re doing - you really should create a new branch for this but it’s super minor) then pushing that file to staging area and committing just that can save some time

Branch

  • An independent line of development to experiment with a different project version
  • Naming conventions:
  • Something that describes the purpose of the branch
  • No white space

Master branch

  • The main branch of the remote project
  • When you create a branch, new branch and master have the same commit histories up to that point. Commits on the new branch have no impact on master
  • Note: main = master

Merge conflict

  • Git doesn’t know what changes you want to keep
  • If master has a commit that alters the same code as what’s in the branch and then the branch is attempted to be merged
  • Note: VS code will show you the conflict

Remote

  • A shared git repository that allows multiple collaborators to work on the same project from different locations
  • Collaborators work independently and merge changes together when ready
  • Features of remote teamwork:
  • Complete replica of the project on everyone’s computer
  • Way to keep track of and review everyone else’s work
  • Access to a project version

Fast-forward merge

  • When git brings your local master branch up to speed with the most recent commits on the remote. See git fetch for syntax

Typical git workflow

  1. Fetch and merge changes from the remote
  2. Create a branch to work on a new feature
  3. Develop feature on your branch and then commit your work
  4. Fetch and merge from the remote again, in case new commits were made while you were working
  5. Push your branch up to the remote for review by manager

Basic syntax

$git init

  • Initialize your project. Sets up all the tools that Git needs to start

$git status

  • Check contents of working directory

$git add filename1 filename 2

  • Add a file to staging area
  • git add . for everything

$git diff

  • Check differences between working directory and staging area

$git commit

  • Permanently stores staging area changes inside repository
  • Use git commit -a -m to add changes and include message in command line
  • Commit message conventions:
  • Written in present tense
  • Should be 50 characters or less

$git log --oneline

  • See a list of your commits

$git show HEAD

  • See a summary in terminal the HEAD commit, and everything that was done in that commit including the comment

$git checkout HEAD filename

  • Restore your working directory file to look exactly like it did when commit was made
  • This essentially undoes all the changes you’ve done to the previous commit, even if those changes are in the staging area

$git reset HEAD filename

  • Unstage a file from the staging area before committing
  • The staging area will now look like the HEAD commit
  • For example, if you accidentally did something to file 3 unrelated to files 1 and 2. Files 1, 2, and 3 are all in the staging area but you only want to commit 1 and 2, not 3. So get 3 out of there with ths command
  • You can do a $git status afterwards to confirm this

$git reset 5d69206

  • Use the first 7 characters of a SHA of a previous commit
  • HEAD now gets set to that commit you asked for
  • Changes the commit history

$git restore filename

  • Restore non-committed changes - these are changes either in your working copy or your staging area (aka index)
  • So it erases everything that wasn’t committed - i.e. undoes everything not yet in your staging area

Branching syntax

git branch

  • List of branches. Master will always be one

git branch new_branch

  • Create a branch

git checkout new_branch

  • Switch to your new branch

git merge new_branch

  • Include all the changes made on one branch with the master branch
  • Note: you have to do this when on the master branch

git branch -d new_branch

  • Delete a branch

Teamwork syntax

git clone remote_location clone_name

  • Get your own replica of the repository
  • Tells git where to find the repo and in what directory to store it in

git remote -v

  • List all the remotes: their name and location

origin

  • The name of the default remote repository

git fetch

  • See if changes have been made to the remote and then bring them down to your local copy
  • Note: it will not merge changes from remote. Rather, it brings the changes into a remote branch
  • You need to do a git merge origin/master to bring the changes into your local master branch
  • This will update files with recent commits to master by doing a fast-forward merge
  • Note: git pull does both of these things

git push origin your_branch

  • Push your branch up to the remote, origin
  • This allows the branch to be reviewed and merged into the master branch
  • Example: git push -u origin master

git submodule

  • Used to embed a foreign repository into your source tree
  • Add using:
  • git submodule add <url>
  • Fetch new changes using:
  • git submodule update --remote --merge
  • Then do a git pull

Removing and reverting

From staging and commit

  • git rm --cached -r
  • --cached tells it to remove the paths from staging and the index without removing the files themselves
  • -r operates on directories recursively

Revert a remote git repo back to a commit

  • Run the following
  • git reset --hard {{commit id i.e. e8196f7}}
  • git push -f origin master

Note: this applies when branch name = master, remote = origin

Note: This is not advised when other people are working on the project
