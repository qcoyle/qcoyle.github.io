Terms

What is express?

  • A flexible Javascript middleware framework for creating web servers and api’s
  • You can use it to create small projects, from static file servers and JSON API’s, to full production servers

Route

  • The control flow for requests into an api based on the path (whatever comes after the host url) and HTTP verb
  • Express routes require two arguments: a string path and a callback function
  • If no routes match a client request, Express handles sending a 404 error back to the client

Routers

  • Mini versions of express applications that handle route matching, requests, and responses
  • They don’t start a separate server or  use their own ports
  • They go by the standard get(), put(), post(), and delete() routes

Express as middleware

  • It’s software acting between the server receiving the request and sending the response. It’s operating on the boundary of the two
  • From the documentation: “An Express application is essentially a series of middleware function calls.”

General Syntax

Create a server

  • Import express and call express()
  • Give the server a port number from which to listen for new requests. Use listen()

const express = require("express");

const app = express();

const PORT = 4001;

app.listen(PORT, () => {

   console.log(`Server is listening on port ${PORT }`);

});

// Do a GET to http://127.0.0.1:4001/ - an error will be returned. However the connection is no longer refused

Set a node process environment variable for the port

const PORT = process.env.PORT  || 4001; // 4001 if there’s no variable defined

Static files

  • Files that clients download as-is from the server - typically all the HTML, CSS, and JS files that make up the front end
  • By default express does not serve static files. It must be enabled using the below code, where public is the name of the directory containing your static files

app.use(express.static('public' ));

  • One pattern is then to make css and js subdirectories to hold respective files

  • Note that all static files you load will now consider the new public directory as the root

Register a GET

  • Use app.get():

const express = require("express");

const app = express();

const PORT = 4001;

app.listen(PORT, () => {

   console.log(`Server is listening on port ${PORT }`);

});

app.use(express.static('public' ));

const users = [{ user: "tbrady" }, { user: "pmahomes" }];

app.get("/users", (req, res , next) => {

   console.log(req);

   // res.send(users); // Provide any response

   res.json(users); // Sends JSON-formatted response

})

// Returns [{"user":"tbrady"},{"user":"pmahomes"}]

Route parameters

  • Route path wildcards to match any text given in a route path
  • Express takes in route parameters via the : path indicator. Then the actual value given is attached to the req object as req.params. Keys of the object are parameter name and the value is the parameter value:

const users = {

   tbrady: { team: "bucs", age: 39 },

   pmahomes: { team: "chiefs", age: 30 }

}

// Call GET http://127.0.0.1:4001/users/tbrady

app.get("/users/:name", (req, res , next) => {

   console.log(req.params); // { name: "tbrady" }

   res.send(users[req. params.name]);

})

// Returns {"team":"bucs","age":39}

Route parameters order

  • To ease debugging, the best way to order routes is:
  1. Root route (i.e. myapi.com/)
  2. Specific route (i.e. myapi.com/users)
  3. Longer routes to a specific path (i.e. myapi.com/users/:id)
  • To ease debugging, routes to the same path should be together. For example:

GET /users/

POST /users/

GET /users/:id

PUT /users/:id

DELETE /users/:id

GET /tags/

POST /tags/

GET /tags/:id

PUT /tags/:id

DELETE /tags/:id

Set your own status code

  • Use the .status() method and chain other messages.
  • You can make the status number anything
  • For example:

const express = require("express");

const app = express();

const PORT = 4001;

app.listen(PORT, () => {

   console.log(`Server is listening on port ${PORT }`);

});

const rapperNames = { dre: "Andre Young", snoop: "Calvin Broadus Jr.", eminem: "Marshall Mathers III" };

app.get('/rappers/:name', (req, res , next) => {

   const rapperName = rapperNames[req .params.name];

   if (rapperName) {

       res.send(rapperName);

   } else {

       res.status(404).send( 'Rapper not found!' );

   }

});

// http://127.0.0.1:4001/rappers/dre returns Andre Young

// http://127.0.0.1:4001/rappers/tupac returns Rapper not found!

Query strings in express

  • Recall query strings are what comes after the ? character after the path. Separated by &
  • In express they aren’t counted as part of the route path; they get parsed into JavaScript objects
  • Can be accessed by req.query

Register a PUT

  • Use app.put(). Example:

const express = require("express");

const app = express();

const PORT = 4001;

app.listen(PORT, () => {

   console.log(`Server is listening on port ${PORT }`);

});

const bestTeam = {

   "1": { name: "Manchester City" , city: "Manchester" }

};

// PUT /bestTeam/1?name=Arsenal&city=London updates bestTeam

app.put("/bestTeam/:id", (req, res , next) => {

   const bestTeamUpdate = req.query ;

   bestTeam[req.params.id] = bestTeamUpdate;

   res.send(bestTeam[req. params.id]);

});

Register a POST

  • Use app.post()
  • Note that POSTs do not end with a route (since they’re creating new data) but rather a type of resource to be created

const express = require("express");

const app = express();

const PORT = 4001;

app.listen(PORT, () => {

   console.log(`Server is listening on port ${PORT }`);

});

const bestTeam = [{ id: 1, team:  "Arsenal", city: "London" }];

app.get("/bestTeam", (req, res , next) => {

   res.send(bestTeam);

});

app.post('/bestTeam', (req, res , next) => {

   const receivedExpression = createElement('teams' , req.query);

   if (receivedExpression) {

       bestTeam.push(receivedExpression);

       res.status(201).send( receivedExpression);

   } else {

       res.status(400).send();

   }

});

// POST http://127.0.0.1:4001/bestTeam?name=Chelsea returns { "id": 2, "name": "Chelsea", "city": "London" }

// Subsequent GET returns [{ id: 1, team: "Arsenal", city: "London" }, { "id": 2, "name": "Chelsea", "city": "London" }]

const createElement = (elementType, queryArguments) => {

   let expressionIdCounter = 1;

   if (queryArguments.hasOwnProperty('name' ) &&

       queryArguments.hasOwnProperty('city')) {

       let currentId;

       if (elementType === 'teams') {

           expressionIdCounter += 1;

           currentId = expressionIdCounter;

       } else {

           animalIdCounter += 1;

           currentId = animalIdCounter;

       }

       return {

           'id': currentId,

           'name': queryArguments.name ,

           'city': queryArguments.city ,

       };

   } else {

       return false;

   }

};

Register a DELETE

  • Use app.delete()
  • They delete existing data so paths should end with a parameter to say what to delete
  • Note: a 204 No Content status should be a returned if a deletion occurs successfully

const bestTeam = [{ id: 1, team:  "Arsenal", city: "London" }, { id:  2, team: "Spurs", city:  "London" }];

app.delete("/bestTeam/:id", (req, res , next) => {

   idInt = parseInt(req.params.id); // Convert id to integer

   const removeIndex = bestTeam.map (x => x.id).indexOf (idInt); // Find array index for id

   if (removeIndex !== -1) {

       bestTeam.splice(removeIndex, 1 ); // Remove from array at index

       res.status(204).send();

   } else {

       res.status(404).send();

   }

});

// DELETE http://127.0.0.1:4001/bestTeam/2 removes Spurs object. DELETE http://127.0.0.1:4001/bestTeam/3 returns a 404

Routing Syntax

Create a router

  • Note: a router will give you a subset of express methods

const express = require('express');

const app = express();

const myRouter = express.Router();

Mount a router

  • This allows us to use a router in our app. Use:

app.use('/users', monstersRouter);

  • It assumes all routes it will use will have the given path (i.e. /users)

Use a mounted router for GET

  • Similar to using a standard app.get:

const express = require('express');

const app = express();

const cities = {

   '1': {

       name: 'Hong Kong',

       population: 5000000

   },

   '2': {

       name: 'Sydney',

       population: 1000000

   }

}

const citiesRouter = express.Router();

app.use('/cities', citiesRouter);

citiesRouter.get('/:id', (req, res , next) => {

   const city = cities[req. params.id];

   if (city) {

       res.send(city);

   } else {

       res.status(404).send();

   }

});

// GET http://127.0.0.1:4001/cities/1 returns Hong Kong object

const PORT = 4001;

app.listen(PORT, () => {

   console.log(`Server is listening on port ${PORT }`);

});

Organizing router files

  • General convention is to separate routers to different paths and nest them
  • Routers can be nested as many times as necessary
  • It helps with scalability: keep each router in its own file and require them in the main application.
  • For instance, the users, orders, and returns routers of an app would be in different files titled users.js, orders.js, and returns.js respectively inside their own routes directory
  • In these files export the router, i.e. for users.js do module.exports = usersRouter;
  • The application (i.e. main.js) would then look like:

const express = require('express');

const app = express();

const PORT = process.env.PORT  || 4001;

// Use static server to serve the Express Yourself Website

app.use(express.static('public' ));

// Import and mount the expressionsRouter

const usersRouter = require('./routes/users.js');

const ordersRouter = require('./routes/orders.js');

const returnsRouter = require('./routes/returns.js');

app.use('/users', usersRouter);

app.use('/orders', ordersRouter);

app.use('/returns', returnsRouter);

app.listen(PORT, () => {

   console.log(`Server is listening on ${PORT }`);

});

router.param()

  • Adds callbacks to route parameters - usually to check the existence of the request’s data relative to the route parameter
  • Allows us not to have to duplicate parameter-matching code for different requests to the same parameter. It avoids duplication in the first line in:

app.get('/users/:username', (req, res , next) => {

   const user = users[req. params.userName];

   res.send(user);

});

app.get('/users/:username/contactInfo', (req, res, next) => {

   const user = users[req. params.userName];

   const contactInfo = users[user .name].contactInfo;

   res.send(contactInfo);

});

  • It intercepts any request to the handler and allows you to run a check - errors occur we pass them to error-handling middleware. If no errors occur, the handler value is attached as a property of the request object
  • Has a callback in which the 4th argument should be the parameter’s value
  • Note that a : is not provided in the app.param() definition because the actual route parameter is passed in as the id value
  • Example for all requests to a users/:userId path:

app.param('userId', (req, res , next, id) => {

   let userId = Number(id);

   try {

       const found = UserList.find ((user) => {

           return userId === user. id;

       })

       if (found) {

           req.user = found;

           next();

       } else {

           next(new Error('Your user was not found in the manifest'));

       };

   } catch (err) {

       next(err)

   }

});

{mergeParameters: true}

  • A router argument that tells express that the router should have access to parents passed into its parent router
  • Example use in the child router:

// /routers/parent

const express = require("express");

const parentRouter = express.Router();

const childRouter = require("/routers/child");

parentRouter.use("/:id/children", childRouter);

const parents = {

   ...

};

// You also need an app.param call to send id down to child routers 

// /routers/child

const router = require('express'). Router({mergeParams: true});

const children = {

   ...

};

module.exports = router;

Nested routers with parameters

  • Routers invoked within other routers, allowing for the composition of more complex api’s after a parameter (i.e. id) of a parent is given
  • Example:

const express = require("express");

const parentRouter = express.Router();

const childRouter = express.Router({ mergeParams:  true });

// Compose in the child router with the parent

parentRouter.use("/:id/children", childRouter);

parentRouter.get("/", (req, res , next) => {

   res.status(200).send( parents);

   next();

});

parentRouter.param("id", (req, res , next, id) => {

   const parent = getParentById(id );

   req.parent = parent;

   next();

});

// This might be in a separate file called /routers/child

childRouter.get("/", (req, res , next) => {

   res.status(200).send( `Parent ${req.parent} has children ${getChildren (parent)}`);

});

app.use("/parent", parentRouter);

Middleware Syntax

Express middleware

  • Express middleware is any function with (req, res, next) as it’s parameters
  • Note: every express route is middleware. All express routes have access to the request, response, and the next middleware in the stack
  • List of maintained express middleware: https://expressjs.com/en/resources/middleware.html

app.use()

  • Middleware before route handling. For example anything in an app.use((callbackFunction()) will run for all requests before routing and path separation takes place
  • Example to log a message after every request:

app.use((req, res, next) => {

   console.log(`${req. method} Request received`);

   next();

});

next()

  • If called explicitly as the last part of the middleware’s body, it hands off processing of the request to the next middleware in the stack
  • Next is always used, even if it seems repetitive, because it helps control confidential information in the case of users who aren’t allowed to see particular information
  • Control flow can be used to only pass next() to users who are allowed to see the information

Path-specific

  • Providing a path to app.use will execute code for any calls made to that specific path:

app.use('/users', (req, res , next) => {

   console.log('Logs for any requests to the /users path. next() allows access to route handlers');

   next();

});

Path-specific with route parameters

  • Convention is to do all checks in the middleware and then add the desired parameter and object to the req object. For example:

const players = {

   messi: {

       number: 10

   },

   ronaldo: {

       number: 7

   },

   mbappe: {

       number: 7

   },

   benzema: {

       number: 9

   }

};

app.use("/players/:name", (req, res , next) => {

   const name = req.params. name;

   if (!players[name]) {

       console.log('Response Sent');

       return res.status(404 ).send('Player with that name does not exist');

   }

   req.playerName = name;

   req.player = players[name]; // Now all next callbacks have access to this info via the req object

   next();

});

Multiple specific paths

  • app.use() can take in an array of paths
  • Note: it can also take in a regular expression to match paths, path patterns, and array combinations of these
  • This functionality is great for not duplicating code when the same logic is used to process different paths
  • Example from Codecademy:

// Add your code below:

app.use(['/beans/', '/beans/:beanName'], ( req, res, next) => {

   let bodyData = '';

   req.on('data', (data) => {

       bodyData += data;

   });

   req.on('end', () => {

       if (bodyData) {

           req.body = JSON.parse (bodyData);

       }

       next();

   })

});

Non-anonymous middleware functions

  • Middleware can be called via declared functions and then added universally or placed path-specifically among the application
  • For example:

const logging = (req, res, next ) => {

   console.log(req);

   next();

};

app.use(logging);

Middleware functions with multiple callbacks

  • Express methods like app.use(), app.get(), and app.post() can take multiple callbacks, for instance:

const authenticate = (req, res, next ) => {

   // Authentication logic

};

const validateData = (req, res, next ) => {

   // Validation logic

};

const getPlayer = (req, res, next ) => {

   res.status(200).send( getPlayerById(req.params.id));

};

const createPlayer = (req, res, next ) => {

   createPlayerFromRequest(req);

   res.status(201).send();

};

const updatePlayer = (req, res, next ) => {

   updatePlayerFromRequest(req);

   res.status(204).send();

}

app.get('/players/:id', authenticate, getPlayer );

app.post('/players', authenticate, validateData , createPlayer);

app.put('/players/:id', authenticate, validateData , updatePlayer);

morgan

  • An open-source library for logging HTTP request and response information
  • Its morgan() function returns a middleware function (the return value of it will have the signature (req, res, next)) that takes in an argument to describe logging output format (i.e. morgan(“tiny”))
  • It gets called before all middleware functions that follow it
  • To use it on every call, require and add

app.use(morgan("dev"));

  • It will color code it too!
  • To use different levels on certain environments, do something like:

if (!process.env.IS_TEST_ENV) {

   app.use(morgan('short'));

}

body-parser

  • Open source middleware that allows us to parse bodies sent with an HTTP request (i.e. for POST and PUT)
  • To use, require it and add:

app.use(bodyParser.json());

  • Note: the above call will parse any sent bodies in JSON format

Error-handling middleware - anticipated errors

  • Anticipated errors can be caught by passing an Error object as an argument to next(). When next() is called with an error as the first argument, it will call error handling middleware
  • Example: note the below code sends a response back to the user but other actions could be taken including logging it, trying again, and sending an error alarm email

app.use((req, res, next) => {

   const newValue = possiblyProblematicOperation();

   if (newValue === undefined) {

       let undefinedError = new Error ('newValue was not defined!');

       undefinedError.status = 404;

       return next(undefinedError);

   }

   next();

});

Error-handling middleware - unanticipated errors

  • Express has its own error object, err, that can be given as a parameter in a callback function
  • We can use information in this object to perform different tasks and send different messages
  • Note: unanticipated error-handling middleware should be the last app.use() functions in your file so that it doesn’t ever get missed. This is because the middleware stack progresses in order
  • Example:

app.use((err, req, res, next) => {

   const status = err.status  || 500;

   res.status(status).send( err.message);

});

errorhandler

  • Middleware library that does what the above code does
  • Require it and call on the last line of file (before app.listen()) using:

app.use(errorhandler());
