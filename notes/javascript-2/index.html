Terminology

React JS

  • Open source JS library for building single-page user interfaces with interaction
  • It seems like it builds in the html to the javascript
  • Works for Android/iOS devices
  • Supported by Facebook

Angular JS

  • A newer, open source JS library for single page applications
  • Based on MVC (Model View Controller) instead of a DOM like React. Furthermore, Angular is based on Typescript while React is based on Javascript
  • Provides testing + debugging for project built in. React requires a set of tools to do this
  • Better for larger scale feature-rich applications, and when development team has experience with Java, C#, and Angular (vs. CSS/HTML/Javascript)
  • Said to be the best React JS alternative
  • Steeper learning curve
  • Supported by Google

How does Javascript run?

  • It’s single-threaded, non-blocking, asynchronous, and concurrent
  • It has a call stack, event loop, and a callback queue

Heap

  • Where memory allocation happens - you don’t really need to worry about it

Call stack

  • Data structure that keeps the information about active subroutines going on in a computer program
  • If we step into a function, we push something onto the top of stack. If returning from a function (return or an implicit return by getting to the end of a function), we pop off the top of it
  • This is the stack trace. When you get an error, you get the stack trace printed out in the order it ran
  • Blowing a stack: Stack gets overfilled, usually because of a bug in which something gets called recursively to infinity. So your engine kills everything and returns the error
  • Also known as execution stack, program stack, control stack, run-time stack, machine stack, or just the stack

Blocking

  • Processes that are slow and on the stack. console.log is not slow. Graphics, network requests, huge while loops, etc. are slow

Single threaded

  • Can do one thing at a time. Only one piece of code can be run at a time
  • This means that blocking programs hold up the call stack if the program is single threaded
  • This is a problem with browsers, because the browser becomes stuck when a blocking program is going on. This doesn’t provide a good user experience. The solution is asynchronous callbacks

Asynchronous operation

  • One that allows the computer to move on to other tasks while waiting for a synchronous operation to be done
  • Example: While waiting for dishwasher to finish dishes, we’re free to work on other chores

Callback

  • Can be one of two things
  • Any function that another function calls
  • A call that will get pushed back onto the callback queue (asynchronous callback)

Asynchronous callbacks

  • Example: when using setTimeout() and stuff below it runs first
  • In node and the browser, there are almost never any blocking functions. This is because calls are made asynchronously

Concurrency

  • Two or more procedures being executed at the same time on the same shared resource
  • The reason we can do things concurrently is that the browser is more than just the runtime (which can only do one thing at a time). For example, the Chrome web API’s are effectively separate threads
  • Example: when we call setTimeout() (which recall, is an API provided to us by the browser) it kicks off a timer in the browsers separately. After the web API is done, it pushes your call onto the task queue

Task queue

  • Where asynchronous callbacks get pushed before going to the stack
  • Controlled by the event loop
  • Aka the event queue

Event loop

  • In a browser, it allows synchronous code to run before any asynchronous code can be run
  • It looks at the stack and the task queue. If the stack is empty, it takes the first callback on the task queue and pushes it back on the stack (i.e. the V8 engine) and the callback finally runs
  • For example, this is why if you call setTimeout(function, 0), it defers operation until the stack is clear

Screen render and callbacks

  • The screen in a browser will only render when the stack is clear
  • While a slow synchronous loop is taking place, the screen cannot render
  • When the stack clears, the screen will render before any callbacks take place
  • “Don’t block the event loop” means don’t put slow code on the stack because then the browser can’t do what it’s meant to do in creating a fluid UI
  • Example: Image process and animation gets slow unless you queue the code appropriately

Javascript as imperative programming

  • See https://quinncoyle.com/notes/software-engineering for programming paradigm definitions
  • I.e. a function that says return the sum of all items in the input array. You create a result variable (let result = 0). Then you make a for loop and sum up all those values + return it.
  • You’re explicitly iterating over an array
  • We’re continually mutating a state variable (result variable)

function add(array) {

   let result = 0;

   for (i in array) {

       result = result + array[i];

   }

   return (result);

}

  • Example using jQuery to modify a web page. Click event handler adds a highlight and changes a text when a button is clicked
  • While you don’t mutate a variable (like result in the last example) you’re continuously modifying the DOM which is part of the state

$("#btn").click(function () {

   $(this).toggleClass("highlight");

   $(this).text() === "Add Highlight"

       ? $(this).text("Remove Highlight")

       : $(this).text("Add Highlight");

});

  • When Javascript is imperative, it’s not quite as readable as declarative. Your brain still has to step through the code to figure out what’s going on

Javascript as declarative programming

  • Javascript can be functional programming, which is a subset of declarative programming
  • Lots of built-in methods like .map, .reduce, and .filter
  • I.e. the below code using JS’s built in reduce method

function add(arr) {

   return arr.reduce((prev, current) => prev + current, 0);

}

  • I.e. using React instead of jQuery:

<Btn

 onToggleHighlight={this.handleToggleHighlight}

 highlight={this.state.highlight}>

   {this.state.buttonText}

</Btn>

  • Note that the state lives in the React component instead of the DOM. This makes it declarative

Hoisting Terminology

Hoisting

  • The way in which the JS engine allocates memory for variable/function names: it raises variable names and function declarations to the top of their scrope and immediately initializes function declarations
  • Initialization (assignment of values to the variable or function) doesn’t take place until code execution
  • This is what allows you to call a function on a line of code before the function is declared (unlike in python)

Hoisting of function declarations

  • They get hoisted to the top of scope
  • Global declaration = top of global scope
  • Local declaration = top of local scope
  • Nested functions: top of the enclosing function scope

Hoisting of var

  • Get hoisted to the top of function blocks
  • When they get hoisted, they’re initialized as undefined until they’re run
  • They don’t get hoisted to the top of any other types of blocks
  • Example:

console.log(myVarVariable); // Returns undefined

var myVarVariable = 1;

Hoisting of let and const

  • Hoisted to the top of their parent block for scope
  • They are not initialized when they’re hoisted. So calling a variable in your code before its initialized results in a reference error
  • This is preferable to var because and error is better than andetected bug (usually you need unit tests to catch undefined values)
  • Example:

console.log(myVarVariable); // Throws reference error

let myVarVariable = 1;

Hoisting of function expressions

  • Follow the rules of variable hoisting - same rules as var, let, and const

Promises Terminology

Promise

  • Objects that represent the outcome of an asynchronous operation
  • Initial state is pending

3 states of a promise

  • Pending: not yet completed
  • Fulfilled: Has completed successfully with a resolved value
  • Example value might be a JSON object
  • Rejected: Failed - usually some sort of error

Settled

  • If a promise is no longer pending. Fulfilled or rejected

Executor function

  • A function parameter of the Promise that usually starts an asynchronous operation, dictating how the promise should be settled
  • Two function parameters:
  • resolve()
  • reject()
  • Example:

const executorFunction = (resolve, reject) => {if/else or some other code in here};

const myFirstPromise = new Promise(executorFunction););

Consume

  • You consume promises - handle returned Promise objects

Separation of concerns

  • Organize code into distinct sections that handle a specific task

Composition

  • The process of chaining promises together - i.e. multiple .then’s followed by a .catch
  • Makes code much more readable than nested callback syntax
  • Use composition if the order does matter

Concurrency

  • Multiple asynchronous operations happening together

Promises syntax

new Promise(executorFunction)

  • Promise constructor method
  • Can name this to a variable

resolve()

  • Changes a promise’s status from pending to fulfilled if called
  • Only takes in one argument
  • The resolved value will then be set to the argument passed into it

reject()

  • Function that changes a promise’s status from pending to rejected
  • Only takes one argument
  • Rejection reason will be set to argument passed into it

Promise.then()

  • Tells what you want to happen after the promise is settled
  • Takes two callback functions (aka two handlers) as arguments:
  • Success handler: onFulfilled
  • Failure handler: onRejected
  • Note: you can chain .then()’s with a failure handler to a first one with a success handler:

prom

   .then((resolvedValue) => {

       console.log(resolvedValue);

   })

   .then(null, (rejectionReason) => {

       console.log(rejectionReason);

   })

  • This chains two .then ‘s to a single promise instance
  • A couple things to remember:
  • Make sure not to nest the logic for one promise inside another
  • Make sure you return a promise anytime you’re handling a resolved value. JS won’t throw an error if you forget this

Promise.catch()

  • Takes in an argument for a rejected promise to invoke the failure handler
  • It’s an extension of the above example of chaining .then() with a failure handler. It allows us to separate resolved logic from rejected logic. This makes it easier to read
  • Example:

prom

   .then((resolvedValue) => {

       console.log(resolvedValue);

   })

   .catch((rejectionReason) => {

       console.log(rejectionReason);

   })

Promise.all()

  • Allows for concurrent execution of promises where the order does not matter (composition)
  • Input: an array of promises
  • Output: a single promise:
  • If every promise in the input resolves, it returns an array with the resolved value from each promise in the input
  • If any promise from the input rejects, it rejects and gives reason (it “fails fast”)
  • Example:

let myPromises = Promise.all([returnsPromOne(), returnsPromTwo(), returnsPromThree()]);

myPromises

 .then((arrayOfValues) => {

   console.log(arrayOfValues);

 })

 .catch((rejectionReason) => {

   console.log(rejectionReason);

 })

Asynchronous Syntax

setTimeout()

  • Add a delay to a function, i.e. setTimeout(delayedHello, 2000);
  • Other functions, even those below it in the code, will run asynchronously before the above function then runs

setInterval()

  • Execute a code block at a specific interval
  • It will continue to execute until clearInterval() is called or until the process exits

const showAlert = () => {

 // Calling setInterval() and passing a function that shows an alert every 5 seconds.

 setInterval(() => {

   alert('I show every 5 seconds!')

 }, 5000);

};

// Calling the newInterval() function that calls the setInterval

showAlert();

async...await

  • Syntax introduced in ES8 that allows the writing of clean asynchronous code using promises and generators
  • It’s syntactic sugar
  • Previously, callback functions were required for asynchronous and dependent code. They encouraged complex nested code which is hard to read and scale. This syntax makes asynchronous code look synchronous, which is much easier to read

async

  • Used to write functions that handle asynchronous actions
  • Asynchronous logic goes inside an async function
  • They always return a promise, so you can use .then() and .catch with async function results
  • Via declaration:

async function myFunc() {

   // Function body here

};

  • Via expression:

const myFunc = async () => {

   // Function body here

};

Three ways an async function returns

  • If nothing is returned → a promise with a resolved value of undefined
  • If a value is returned → a promise resolved to that value
  • Example:

async function fivePromise() {

   return 5;

}

fivePromise()

   .then(resolvedValue => {

       console.log(resolvedValue);

   })  // Prints 5

  • It a promise is returned → that promise

await

  • An operator that returns the resolved value of a promise
  • It pauses the execution of the async function (everything that’s inside the function) until a promise is resolved
  • This makes the promise read like synchronous code
  • It can only be used inside of an async function
  • Example:

async function asyncFuncExample() {

   let resolvedValue = await myPromise();

   console.log(resolvedValue);

}

asyncFuncExample();

  • Note: forgetting the await keyword can be tricky to debug since the function still runs, but a promise object is returned instead of the resolved value

Handling dependent promises

  • Common use case: network requests resulting from a db query

async function asyncAwaitVersion() {

   let firstValue = await returnsFirstPromise();

   console.log(firstValue);

   let secondValue = await returnsSecondPromise(firstValue);

   console.log(secondValue);

}

try...catch

  • Use try...catch
  • Unlike with .catch() in a long promise chain, this tells where in the chain the error was thrown

async function usingTryCatch() {

   try {

       let resolveValue = await asyncFunction('thing that will fail');

       let secondValue = await secondAsyncFunction(resolveValue);

   } catch (err) {

       console.log(err);

   }

}

Using .catch() with async...await

  • While try...catch is preferred, sometimes .catch() is used in the global scope to catch any final errors as in the below example. This is more common with complex code

async function usingPromiseCatch() {

   let resolveValue = await asyncFunction('thing that will fail');

}

let rejectedPromise = usingPromiseCatch();

rejectedPromise.catch((rejectValue) => {

   console.log(rejectValue);

})

Independent processes

  • For processes that can be run concurrently, don’t await the promise-returning functions inside the async function. Instead, await when you need the result:

async function concurrent() {

   const firstPromise = firstAsyncThing();

   const secondPromise = secondAsyncThing();

   console.log(await firstPromise, await secondPromise);

}

await Promise.all()

  • Takes in an array of promises and returns a single promise, which resolves when all promises in the array have resolved
  • The return promise’s resolved value will be an array of resolved values
  • All of the promises will process concurrently
  • As soon as any one of the promises in the array fails, the single promise will reject. This makes it “fail fast”
  • This is another way to handle independent asynchronous promises that don’t depend on each other
  • Example:

async function asyncPromAll() {

   const resultArray = await Promise.all([asyncTask1(), asyncTask2(), asyncTask3(), asyncTask4()]);

   for (let i = 0; i < resultArray.length; i++) {

       console.log(resultArray[i]);

   }

}

Requests Terminology

JSON

  • The de facto standard to store and send data between all programming languages
  • Commonly used for data transfer between browser and server (i.e. a web form)

JSON Syntax

  • Curly braces hold objects
  • Square brackets hold arrays
  • Data is stored in key value pairs separated by colons
  • Pairs are separated by columns
  • Property names must be in quotes (note this is not required for JS object names)
  • Data types must be: string, number (int or float), object (name-value pair), array (separated by commas), boolean, or null
  • Other data types are often converted to strings and follow standards. I.e. 2014-01-01T23:28:56.782Z for dates

AJAX

  • A development technique (Asynchronous JavaScript and XML) to build asynchronous web applications by accessing servers via http
  • It’s the use of the XMLHttpRequest object to access servers
  • Note that AJAX can be misleading, as plain text and JSON text are also used to transport data
  • Powerful because it lets pages be updated asynchronously without reloading the page or freezing the browser window

XMLHttpRequest (XHR)

  • A Javascript api that creates objects to interact with servers
  • Like AJAX, it’s name is misleading. Supports text and JSON in addition to XML

Fetch API

  • Interface for network requests - a more powerful and flexible extension of XMLHttpRequest

Requests Syntax

JSON.stringify()

  • Converts a Javascript value to a JSON string which can then be sent to a server

?

  • Denotes a query string: additional information sent with an HTTP request

'https://api.datamuse.com/words?key=value'

&

  • Used to combine multiple query strings

'https://api.datamuse.com/words?key=value&anotherKey=anotherValue'

fetch()

  • Does 3 things:
  1. Creates a request object with information used when sending to an api
  2. Sends the request object to an API endpoint you provide
  3. Returns a promise that resolves to a response object, containing status of the promise along with information sent back by the API

Example Fetch API GET Request using .then() (AJAX)

fetch('https://api-to-call.com/endpoint').then(response => {

   if (response.ok) {

       return response.json();

   }

   throw new Error('Request failed!');

}, networkError => {

   console.log(networkError.message);

}).then(jsonResponse => {

   return jsonResponse;

   myFunction(jsonResponse);

});

  • Returning response.json allows whatever function is next chained to the fetch (using .then()) to receive a promise with the JSON data inside
  • Two functions are passed after the first .then(): the first function handles a resolved promise, the second handles if the promise is rejected
  • The second .then() is chained to the first one
  • myFunction is a function that takes in the JSON response from the API and manipulates it

Currying in Javascript

Example of a non-curried function

function add(a, b) {

   return a + b;

}

  • Note: if add() is called with only one argument, it will still run and be evaluated as 1+undefined, returning NaN

Closure

  • When a nested function keeps the scope of parent functions, so they have access to any variables declared in the scope of parent functions
  • Access is kept even after parent functions are finished executing

Example of a curried function

function curried_add(a) {

   // has access to the argument for a

   return function nested_add(b) {

       // has access to the arguments for a and b

       return a + b;

   }

}

// Example 1

console.log(curried_add(5)(10));

// Returns 15

// Example 2

let add_one = curried_add(1); // Creates a local variable a and assigns it the value 1

console.log(add_one(10)); // Add_one() still has access to the argument from curried_add()

// Returns 11

  • curried_add() returns another function named nested_func()
  • So when you call add(a), nested_func(b) is returned and the output will then be evaluated as a+b as desired
  • The nested function gets access to the second variable because of closure

Currying with arrow functions

  • Makes things much more concise
  • Same example as above:
  • let curried_add = a => b => a + b;
